## A. Google Search Console Setup Checklist

### A1. Property Setup and Verification

- [ ] **Go to**: https://search.google.com/search-console/welcome
- [ ] **Add property** as **Domain property** (covers http/https and all subdomains): `moltyflywheel.com`
- [ ] **Choose verification method** — recommended: DNS TXT record
  - Log in to your DNS provider (Cloudflare)
  - Add TXT record: `google-site-verification=XXXXXXXX` to root domain
  - Return to Search Console and click Verify
  - _Alternative_: HTML file method — place verification file in `public/` directory so it is served at `https://moltyflywheel.com/google-site-verification.html`
- [ ] **Confirm property is verified** before proceeding to next steps

### A2. Sitemap Submission

- [ ] Sitemap URL to submit: `https://moltyflywheel.com/sitemap-index.xml`
  - This is generated by `@astrojs/sitemap` at build time
  - robots.txt at `https://moltyflywheel.com/robots.txt` already references this URL
- [ ] In Search Console → **Sitemaps** → enter `sitemap-index.xml` → click Submit
- [ ] **Check returned status** — should show "Success" with a page count ~67
  - If count is low, check whether the last build was deployed
  - If sitemap returns 404, confirm Cloudflare Pages has the latest build deployed

### A3. Indexing Verification After Submission

- [ ] In Search Console → **URL Inspection** → enter: `https://moltyflywheel.com`
  - Expect: "URL is on Google" after crawl, or "URL is not on Google" (means not yet indexed)
  - If not indexed: click **Request Indexing**
- [ ] Spot-check key pages via URL Inspection:
  - `https://moltyflywheel.com/blog`
  - `https://moltyflywheel.com/programs`
  - `https://moltyflywheel.com/offers`
  - One live blog post (e.g., `/blog/what-is-agentic-ai-2026`)
  - One program page (e.g., `/programs/koala-writer`)
  - One offer page (e.g., `/offers/best-affiliate-programs-for-beginners`)
- [ ] **Request Indexing** for any pages not yet on Google

### A4. Coverage Report Review

- [ ] In Search Console → **Pages (Coverage)** — review after first crawl (may take 3–7 days)
- [ ] Flag any pages showing "Crawled — currently not indexed" or "Discovered — currently not indexed"
- [ ] Watch for any **Excluded** pages:
  - `/privacy` and `/newsletter` should appear as "Excluded by noindex" or filtered by robots — confirm these match the sitemap filter applied in `astro.config.mjs`
  - `/go/*` redirect paths should not appear as indexed pages (they are 301s to external URLs)
- [ ] Check that **all 7 static offer pages** appear in coverage (they are not in a content collection — confirm sitemap picks them up as static routes)

### A5. Canonical and Duplicate Review

- [ ] Confirm no canonical self-pointing errors reported in Search Console
- [ ] Confirm `/p/[offer]` pages and `/offers/[page]` pages are not flagged as duplicates of each other
  - These serve different content in different templates — they should be treated as distinct
- [ ] Check for any "Alternate page with proper canonical tag" notices (indicates Google found a duplicate and deferred to the canonical)

### A6. Route Families to Inspect First in Search Console

Priority order when reviewing coverage post-submission:

1. **Blog detail pages** (`/blog/[slug]`) — highest content volume, primary SEO asset
2. **Program pages** (`/programs/[program]`) — 20 pages, decision-oriented — should rank for tool queries
3. **Offer hub + static offer pages** (`/offers/`) — 8 pages total
4. **Homepage** (`/`) and hub pages (`/blog`, `/programs`)
5. **Content-driven offer pages** (`/p/[offer]`) — 2 pages, lower priority right now
6. **Niche pages** (`/niche/[niche]`) — 6 pages, check after fixing `/niches` internal link issue

### A7. Pre-Submission Bug to Fix

- [ ] **Fix internal links pointing to `/niches/` (plural)** — current live route is `/niche/[slug]` (singular)
  - `CODEBASE_MEMORY.md` flags this: some offer pages link to `/niches` which 404s
  - Find all `href="/niches` in `src/pages/offers/` and correct to `/niche/`
  - Do this before submitting to Search Console to avoid crawl errors on first pass

## B. Analytics Setup and Validation Checklist

### B1. Confirm Plausible Server Is Live

- [ ] Open browser → navigate to: `https://plausible.moltyflywheel.com`
  - If you see the Plausible login screen, the server is running
  - If you see a connection error or 404, the server is down — no data is being captured
- [ ] Log in to Plausible → confirm `moltyflywheel.com` exists as a tracked site
- [ ] Check **Last 30 days** — confirm there is any data at all
  - If zero visitors despite traffic, script is not loading or server is unreachable from production

### B2. Validate Pageview Script Is Loading

- [ ] Go to live site: `https://moltyflywheel.com`
- [ ] Open browser DevTools → **Network** tab → filter by `script.js`
  - Confirm `script.js` loads from `https://plausible.moltyflywheel.com/js/script.js`
  - HTTP status should be 200
  - If status is 404 or connection refused: server is unreachable from public internet
- [ ] With DevTools open on **Network** → navigate between pages
  - Confirm a `POST` request fires to `plausible.moltyflywheel.com/api/event` for each page load
  - If no POST fires: script loaded but server rejected or blocked the event

### B3. Route Coverage Validation

Confirm Plausible is receiving pageviews from all major route families:

- [ ] Visit `https://moltyflywheel.com` → check Plausible dashboard for homepage visit
- [ ] Visit any blog post → check Plausible for `/blog/[slug]` entry
- [ ] Visit any program page → check for `/programs/[program]` entry
- [ ] Visit any offer page → check for `/offers/[page]` entry
- [ ] Visit any `/p/` page → check for `/p/[offer]` entry

> Pages using `BaseLayout` (all except any using a layout that does not inherit from it): all should auto-track.
> Verify `OfferArticleLayout.astro` — confirm it wraps `BaseLayout` or includes the Plausible script directly.

### B4. Ad Blocker Consideration

- [ ] Note: self-hosted Plausible on a custom subdomain reduces (but does not eliminate) ad-blocker filtering
- [ ] If traffic looks implausibly low, test with an ad blocker disabled and compare
- [ ] Consider this when interpreting measurement data

### B5. What NOT to Build Now

- Do not set up GA4 in parallel unless there is a specific reason — dual analytics creates noise
- Do not set up a Tag Manager layer until the event scope is defined and approved
- Do not implement server-side analytics until basic client-side verification passes

## C. Event Tracking Checklist

### C1. Prerequisite: Switch to the Tagged Events Script

The current script in `BaseLayout.astro`:
```html
<script defer data-domain="moltyflywheel.com" src="https://plausible.moltyflywheel.com/js/script.js"></script>
```

Must be changed to the outbound links + tagged events variant before any custom events work:
```html
<script defer data-domain="moltyflywheel.com" src="https://plausible.moltyflywheel.com/js/script.outbound-links.tagged-events.js"></script>
```

> In self-hosted Plausible, this combined script file must be enabled in your server config/dashboard.
> Using `script.outbound-links.js` variant will auto-track outbound clicks as a special Plausible goal — this alone will capture affiliate clicks at the `/go/*` URL level.
> Using `tagged-events.js` additionally allows custom `plausible('EventName')` calls.

- [ ] Confirm self-hosted Plausible supports the combined script variant
- [ ] Swap script source in `BaseLayout.astro`
- [ ] Verify new script loads on production without 404

### C2. Minimum Viable Event Layer (implement in priority order)

#### C2a. Outbound Affiliate Click Tracking

If the `outbound-links` script variant is enabled, Plausible will automatically track any click on an `<a>` tag pointing to an external domain as an "Outbound Link: Click" event with the destination URL as the property.

- [ ] Enable `script.outbound-links.js` variant on Plausible server
- [ ] Confirm `/go/*` links (which redirect externally) are captured as outbound clicks
- [ ] In Plausible dashboard → create a **Goal** for "Outbound Link: Click" to surface this data

This alone will report:
- Which `/go/` redirect was clicked
- Which page the click came from

#### C2b. AffiliateButton Click — Labeled Events

Add a named event to `AffiliateButton.astro` so you can filter by program:

```astro
<!-- AffiliateButton.astro — add onclick -->
<a
  href={href}
  onclick={`plausible('Affiliate Click', {props: {program: '${program}'}})`}
  rel="nofollow sponsored"
  target="_blank"
  ...
>
```

- [ ] Add `data-event-name="Affiliate Click"` or onclick event call to `AffiliateButton.astro`
- [ ] Confirm event fires in Plausible dashboard under Goals
- [ ] Named property: `program` = the program ID (e.g., `koala-writer`, `surfer-seo`)

#### C2c. Blog → Program/Offer CTA Click

BlogLayout inline bottom CTA is a raw `<a href="/go/...">` tag:

- [ ] Add click tracking to the inline CTA in `BlogLayout.astro` (line ~173)
- [ ] Suggested event: `plausible('Blog CTA Click', {props: {program: '[program-id]', source: 'blog-bottom-cta'}})`

#### C2d. Advertiser Referral Click

`/go/advertiser` redirects to `https://affitor.com/join/advertiser?aff=REF-93D7EA-E3FF`

- [ ] Ensure this specific click is tracked — either via outbound auto-tracking or named event
- [ ] Suggested event: `plausible('Advertiser Referral Click')` on any CTA pointing to `/go/advertiser`

#### C2e. Offer Page CTA Click

Static offer pages use `AffiliateButton` — covered by C2b above if implemented.
Content-driven `/p/` pages also use `AffiliateButton` — covered.

- [ ] Confirm offer page CTA events appear in Plausible with `program` property after C2b is done

#### C2f. "Start / Apply / Explore" Freeform CTAs

Any CTA that is not `AffiliateButton` (e.g., "Get the Free Guide", "Browse Programs"):

- [ ] Identify all non-affiliate CTAs in `BlogLayout.astro`, `BaseLayout.astro`, and key static pages
- [ ] Add tracking only to CTAs that indicate conversion intent — skip nav links and breadcrumbs

### C3. Suggested Event Naming Convention

| Action | Event Name | Props |
|---|---|---|
| Affiliate button click | `Affiliate Click` | `program: [id]`, `source: [page-type]` |
| Blog inline CTA click | `Blog CTA Click` | `program: [id]`, `source: blog-bottom-cta` |
| Outbound link click (auto) | `Outbound Link: Click` | auto-set by Plausible |
| Advertiser referral click | `Advertiser Referral Click` | `source: [page-slug]` |
| Free guide / newsletter CTA | `Newsletter CTA Click` | `source: [page-slug]` |
| Program page CTA click | `Program CTA Click` | `program: [id]` |

**Convention rules:**
- Event names: Title Case, short, action-oriented
- Props: lowercase, hyphen-separated where multi-word
- Do not encode page URLs into the event name — use `source` prop for that
- Keep event count low — 6–8 events is enough for this stage

### C4. What NOT to Overbuild Now

- Do not implement scroll depth tracking yet — not needed until content length strategy is validated
- Do not implement form submission tracking (no forms beyond the newsletter, which redirects externally)
- Do not set up funnel visualization yet — capture raw event data first, funnels come after 30+ days of data
- Do not add session replay (e.g., Hotjar) until pageview data shows sufficient volume to make it worth it
- Do not build a custom dashboard — Plausible's built-in view is sufficient for the current measurement scope
